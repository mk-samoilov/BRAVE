# Модуль brv_assets

## Подключение плагина

```rust
game.add_plugin(AssetPlugin::new("assets/", "my_xor_key_123"));
//                                 ^путь к папке    ^ключ шифрования astdb.lock
```

Первый аргумент — путь к папке с ассетами (используется в debug-режиме). Второй — строка-ключ для XOR-шифрования `astdb.lock` (используется в release).

---

## API загрузки

Один метод `load` с enum `AssetType`:

```rust
pub enum AssetType {
    GLTFModel,   // папка с scene.gltf + textures/ (многофайловый)
    GLBModel,    // одиночный .glb файл
    Texture,     // PNG/JPG/HDR
    Shader,      // GLSL (компилируется в SPIR-V)
}
```

### Правило именования

- **GLTF модели (многофайловые)** — путь до папки, без расширения
- **Всё остальное (однофайловое)** — полный путь с расширением

```rust
// GLTF — путь до папки (внутри ожидается scene.gltf + textures/)
game.assets.load("models/chair", AssetType::GLTFModel)

// GLB — полный путь с расширением (один файл)
game.assets.load("models/weapon.glb", AssetType::GLBModel)

// Текстуры — полный путь с расширением
game.assets.load("textures/grass.png", AssetType::Texture)

// Шейдеры — полный путь с расширением
game.assets.load("shaders/mesh.vert.glsl", AssetType::Shader)
```

Движку не важна структура папок — модель может лежать в `assets/audio/`, текстура в `assets/models/`. Путь — просто строковый ключ для идентификации ассета.

### Привязка к объекту

```rust
let mesh = game.assets.load("models/chair", AssetType::GLTFModel);
player.mesh.set(mesh);
```

## LRU кеш

Загруженные ассеты кешируются в памяти. Стратегия LRU (Least Recently Used): при достижении лимита вытесняется ассет, к которому дольше всего не обращались.

```rust
game.assets.set_cache_limit(1024)    // лимит в МБ (по умолчанию 512)
game.assets.get_cache_limit() -> u64 // текущий лимит
```

Модели (vertex buffers, текстуры) хранятся в видеопамяти GPU. Лимит кеша считается по GPU-памяти. При вытеснении GPU-ресурсы освобождаются.

При повторном вызове `load()` с тем же путём — ассет берётся из кеша, без чтения с диска.

## Два режима работы

### Debug (`cargo run`)

Ассеты читаются напрямую из папки `assets/`. Файлы парсятся и конвертируются в рантайме:

1. `load("models/chair", GLTFModel)` → открывает `assets/models/chair/scene.gltf`
2. Парсит GLTF, загружает текстуры из `textures/` внутри папки
3. Конвертирует в vertex/index buffers + RGBA текстуры
4. Upload на GPU
5. Добавляет в LRU кеш

Никаких `.ast` файлов. Никакого шифрования. Удобно для разработки — изменил текстуру, перезапустил игру, увидел результат.

### Release (`cargo build --release`)

Ассеты читаются из скомпилированных `.ast` файлов. Данные уже в финальном формате — загрузка сводится к seek + read + upload на GPU. Никакого парсинга в рантайме.

1. При старте — расшифровывает `astdb.lock` XOR-ключом
2. Парсит TOML, строит `HashMap<String, AssetEntry>` в памяти
3. `load("models/chair", GLTFModel)` → ищет `"models/chair"` в HashMap
4. Открывает нужный `x64_{буква}.ast`, seek на offset, read N байт
5. Данные уже готовы — сразу upload на GPU
6. Добавляет в LRU кеш

## Формат .ast

### Компиляция

При `cargo build --release` срабатывает `build.rs` (в крейте brv_assets), который:

1. Обходит все файлы в `assets/`
2. Конвертирует каждый ассет в финальный формат:
    - **Текстуры** → raw RGBA пиксели (декодированные из PNG/JPG/HDR)
    - **Шейдеры** → SPIR-V байткод (скомпилированные из GLSL)
    - **GLTF-папки** → один блоб: vertex buffers + index buffers + RGBA текстуры склеены в непрерывный кусок. Вся папка (scene.gltf + textures/) конвертируется в один blob.
    - **GLB файлы** → аналогично, один блоб
3. Пакует конвертированные данные подряд в `.ast` файлы

### Разбиение на файлы

Каждый `.ast` файл заполняется до ~2.5 ГБ. Если следующий ассет целиком не влезает в текущий файл — он целиком переходит в следующий файл. Ассет **никогда не разрезается** между файлами.

Именование файлов: `x64_a.ast`, `x64_b.ast`, ... `x64_z.ast`, затем `x64_a0.ast`, `x64_b0.ast`, ... `x64_z0.ast`, затем `x64_a1.ast`, `x64_b1.ast`, ... `x64_z1.ast`, и так далее (`a2..z2`, `a3..z3`, ...).

### Расположение

Все `.ast` файлы и `astdb.lock` кладутся в ту же папку, что и бинарник игры. При запуске release-билда рядом с exe лежат:
```
my-game.exe          (или my-game на Linux)
x64_a.ast
x64_b.ast
astdb.lock
```

## astdb.lock

Индексный файл в формате TOML. Содержит разметку всех ассетов: в каком `.ast` файле лежит каждый ассет и на каких байтах.

Ключ = путь от корня `assets/` (тот же что используется в `load()`). Значение = файл (буква) + начало и конец в байтах.

```toml
[models/chair]
file = "a"
begin = "0"
end = "4523876"

[models/weapon.glb]
file = "a"
begin = "4523876"
end = "8291004"

[textures/grass.png]
file = "b"
begin = "0"
end = "2097152"

[shaders/mesh.vert.glsl]
file = "b"
begin = "2097152"
end = "2098340"
```

### Шифрование

`astdb.lock` шифруется XOR с ключом, заданным в `AssetPlugin::new("assets/", "ключ")`. Тот же ключ используется для расшифровки в рантайме. XOR простой — циклически применяется к каждому байту файла.

Это не криптографическая защита, а обфускация — предотвращает тривиальный просмотр структуры ассетов. Сами `.ast` файлы не шифруются (это бинарные блобы без метаданных, без индекса знать какой ассет где лежит — бесполезны).

## Лоадеры

Каждый формат имеет свой лоадер — модуль, который знает как прочитать конкретный тип файла и конвертировать его в данные для GPU:

- **GltfLoader** (`gltf_loader.rs`) — парсит папку с `scene.gltf` + текстуры. Извлекает vertex positions, normals, UVs, indices, PBR-материалы, текстуры. Крейт: `gltf`
- **GlbLoader** (`glb_loader.rs`) — парсит `.glb` файл (бинарный glTF, всё в одном файле). Крейт: `gltf`
- **ImageLoader** (`image_loader.rs`) — декодирует PNG/JPG/HDR в raw RGBA пиксели. Крейт: `image`
- **ShaderLoader** (`shader_loader.rs`) — компилирует GLSL в SPIR-V байткод. Крейт: `shaderc` или `naga`

В debug-режиме лоадеры работают в рантайме. В release-режиме лоадеры используются один раз при компиляции (build.rs), а в рантайме данные уже готовы.
